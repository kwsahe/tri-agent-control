# CODEX.md

## 1. 프로젝트 개요

이 프로젝트의 이름은 **TriAgent Control**임.

Codex, Antigravity, Claude Code를 하나의 로컬 웹 대시보드에서 실행하고 관제하는 멀티 에이전트 개발 오케스트레이터를 구축하는 것이 목표임.

세 모델은 동일한 프로젝트 문맥을 공유하며 다음 작업을 수행함.

- 프로젝트 구조 분석함
- 서로의 의견을 검토하고 반박함
- 역할과 책임 범위를 토론함
- 사용자 승인 후 실제 코드를 수정함
- 테스트, 린트, 빌드 등 가능한 검증을 실행함
- 다른 모델의 변경사항을 교차 검토함
- 최종 결과와 남은 위험을 사용자에게 보고함

이 프로젝트는 완전 자율 개발 시스템이 아니라, **사용자가 팀장으로 남아 의사결정과 위험한 변경을 통제하는 반자동 개발 관제 도구**를 목표로 함.

## 2. 현재 상태에 대한 냉정한 평가

현재 프로젝트는 아이디어 단계가 아니라 동작 가능한 알파 버전 수준임.

이미 다음 기능이 일정 부분 구현되어 있음.

- Codex, Antigravity, Claude Code CLI 실행함
- 로컬 웹 대시보드 제공함
- 토론 모드와 코딩 모드 제공함
- 세션 저장 및 재개 지원함
- 사용자 승인, 보류, 중단 기능 제공함
- 역할 지정 및 역할별 파일 범위 검사함
- 에이전트 간 호출 기능 제공함
- 프로젝트 변경 감지 및 diff 체크포인트 저장함
- 일부 파일 롤백 기능 제공함
- 테스트, 린트, 빌드 자동 감지 및 실행함
- 세션별 메모리와 대화 압축 기능 제공함
- 토큰과 실행 시간 관제 기능 제공함

현재 수준은 대략 **개인용 알파 버전 0.4~0.6**으로 판단함.

### 실현 가능성 평가

| 평가 항목 | 예상 가능성 | 판단 |
|---|---:|---|
| 기본 아이디어 구현 | 90% | 핵심 구조가 이미 구현됨 |
| 개인용 로컬 도구 완성 | 75% | 충분히 실현 가능함 |
| 실제 프로젝트에 안전하게 적용 | 55~65% | 제한적 조건에서 가능함 |
| 실제 토론과 역할 분담 | 약 60% | 형식은 구현됐지만 품질 보장은 어려움 |
| 장시간 자율 코딩 | 35~45% | 반복 오류와 상태 불일치 위험이 큼 |
| 상용 서비스 수준 | 25~35% | 구조 개편과 보안 강화 없이는 어려움 |

### 현실적인 최종 목표

다음 수준을 목표로 해야 함.

> 내 PC에서 Codex, Antigravity, Claude Code를 연결하고, 세 모델이 프로젝트를 분석·토론·역할 분담한 뒤 사용자 승인하에 순차적으로 코드 수정과 검토를 수행하는 개인용 개발 대시보드

다음 목표는 현재 범위에서 추구하지 않음.

> 사람의 개입 없이 세 AI가 장시간 자율 실행되며 어떤 프로젝트든 안정적으로 완성하는 범용 AI 개발팀

## 3. 최우선 운영 원칙

### 3.1 사용자는 항상 최종 결정권자임

사용자는 이 프로젝트의 팀장이며, 에이전트는 사용자의 명시적 의사결정을 대신하지 않음.

설계 선택, 기능 범위, 위험한 변경, 삭제, 의존성 추가, 비용 증가, 권한 확대처럼 결과가 크게 달라질 수 있는 판단은 반드시 사용자에게 맡김.

### 3.2 필요한 경우 반드시 멈추고 질문함

개발 도중 **사용자의 응답 없이는 안전하거나 정확하게 다음 단계로 진행할 수 없는 경우 즉시 작업을 중단하고 질문해야 함.**

이 규칙은 다른 모든 진행 규칙보다 우선함.

다음 상황에서는 추측하거나 임의로 결정하지 말고 반드시 멈춤.

- 요구사항이 서로 충돌함
- 두 가지 이상의 설계안이 있고 결과 차이가 큼
- 기존 기능을 삭제하거나 동작을 크게 변경해야 함
- 데이터 손실 가능성이 있음
- 파일, 폴더, 데이터베이스 또는 세션을 삭제해야 함
- 기존 API나 저장 형식을 깨뜨릴 가능성이 있음
- 새로운 외부 패키지나 서비스를 추가해야 함
- 유료 API, 토큰 또는 비용이 발생할 수 있음
- 관리자 권한이나 위험한 실행 권한이 필요함
- 보안 정책을 완화해야 함
- 테스트를 통과시키기 위해 테스트 기준을 변경해야 함
- 사용자가 정하지 않은 UI, 제품 정책 또는 운영 정책을 선택해야 함
- 역할 분담 결과가 충돌하며 시스템이 스스로 해결할 근거가 부족함
- 같은 오류가 반복되어 새로운 방향 선택이 필요함
- 예상보다 변경 범위가 크게 증가함
- 프로젝트 외부 파일에 접근하거나 수정해야 함
- 비밀키, 인증정보, 개인정보 또는 민감한 데이터가 발견됨
- Git 히스토리 재작성, 강제 푸시, 브랜치 삭제가 필요함
- 기존 사용자 작업을 덮어쓸 가능성이 있음

### 3.3 질문 시 준수 형식

질문이 필요할 때는 막연하게 질문하지 않음.

반드시 다음 내용을 함께 보고함.

1. 현재까지 확인한 사실
2. 작업이 멈춘 정확한 이유
3. 선택 가능한 대안
4. 각 대안의 장점과 위험
5. 추천하는 기본안
6. 사용자에게 필요한 답변 형식

예시:

```text
작업을 일시 중단함.

현재 세션 저장 형식을 변경해야 다음 기능을 구현할 수 있음.
기존 JSON 형식을 유지하면 호환성은 보장되지만 구현이 복잡해짐.
새 형식으로 교체하면 구조는 단순해지지만 기존 세션 마이그레이션이 필요함.

선택지:
A. 기존 형식 유지
B. 자동 마이그레이션 추가 후 새 형식으로 전환
C. 기존 세션 호환을 포기하고 새 형식으로 전환

추천안: B
진행할 선택지를 A, B, C 중 하나로 답변해주기 바람.
```

사용자의 답변을 받기 전까지 관련 변경을 실행하지 않음.

### 3.4 질문 없이 진행 가능한 경우

다음 조건을 모두 만족하면 별도 질문 없이 진행 가능함.

- 사용자 요구사항이 명확함
- 변경 범위가 기존 요청 안에 있음
- 데이터 손실 위험이 없음
- 기존 공개 API와 저장 형식을 깨뜨리지 않음
- 새 비용이나 외부 서비스가 발생하지 않음
- Git으로 쉽게 되돌릴 수 있음
- 테스트 또는 정적 검증으로 결과를 확인할 수 있음

## 4. 개발 우선순위

새로운 기능을 계속 추가하기 전에 다음 세 가지 작업을 우선함.

### 4.1 `roundtable.py` 모듈 분리함

현재 핵심 기능 대부분이 하나의 대형 파일에 집중되어 있음.

이 상태에서 기능을 추가하면 다음 문제가 발생할 가능성이 큼.

- 변경 영향 범위를 파악하기 어려움
- 전역 상태 의존성이 증가함
- 동시 실행 오류를 추적하기 어려움
- 모델별 CLI 변경이 전체 시스템에 영향을 줌
- 테스트와 유지보수가 어려워짐

권장 구조는 다음과 같음.

```text
triagent/
├─ app.py
├─ domain/
│  ├─ session.py
│  ├─ workflow.py
│  ├─ roles.py
│  └─ events.py
├─ agents/
│  ├─ base.py
│  ├─ codex.py
│  ├─ claude.py
│  └─ antigravity.py
├─ orchestration/
│  ├─ coordinator.py
│  ├─ debate.py
│  ├─ coding_loop.py
│  └─ judge.py
├─ workspace/
│  ├─ snapshot.py
│  ├─ diff.py
│  ├─ permissions.py
│  └─ rollback.py
├─ validation/
│  ├─ runner.py
│  └─ result.py
├─ persistence/
│  ├─ session_store.py
│  └─ memory_store.py
└─ web/
   ├─ server.py
   └─ api.py
```

단, 한 번에 전체 파일을 재작성하지 않음.

기존 테스트를 유지하면서 기능 단위로 점진적으로 추출함.

권장 추출 순서는 다음과 같음.

1. 모델별 CLI 어댑터 분리함
2. 프로젝트 스냅샷과 diff 기능 분리함
3. 역할과 권한 정책 분리함
4. 세션 저장소 분리함
5. 워크플로 상태 전이 분리함
6. HTTP 서버와 대시보드 API 분리함

각 단계가 끝날 때마다 기존 테스트와 실제 CLI 연결 확인을 수행함.

### 4.2 독립적인 Judge 구현함

현재 최종 보고 담당 모델은 회의 정리자에 가까우며, 독립적인 판정자로 보기 어려움.

Judge는 다음 원칙을 따라야 함.

- 실제 코드 구현에 참여하지 않음
- 구현자의 설명만 믿지 않음
- diff, 테스트 결과, 정적 분석 결과를 근거로 판단함
- 요구사항 충족 여부를 항목별로 채점함
- 차단 문제가 있으면 승인하지 않음
- 근거가 부족하면 재검증을 요청함
- 결과를 구조화된 형식으로 반환함

권장 출력 형식은 다음과 같음.

```json
{
  "decision": "approve | revise | stop_and_ask_user",
  "scores": {
    "requirements": 0.0,
    "tests": 0.0,
    "security": 0.0,
    "maintainability": 0.0
  },
  "blocking_issues": [],
  "non_blocking_issues": [],
  "evidence": [],
  "next_agent": "",
  "required_action": "",
  "user_question": ""
}
```

`stop_and_ask_user`가 반환되면 자동 진행을 즉시 중단함.

### 4.3 무제한 코딩을 목표 기반 연속 실행으로 변경함

종료 조건이 없는 반복 실행은 사용하지 않음.

기능 이름과 동작을 **목표 기반 연속 실행**으로 변경함.

반드시 다음 제한을 설정함.

- 최대 사이클 수
- 최대 실행 시간
- 최대 토큰 또는 비용
- 최대 변경 파일 수
- 최대 diff 줄 수
- 연속 테스트 실패 제한
- 동일 오류 반복 제한
- 동일 파일 왕복 수정 감지
- 사용자 중단 기능
- 각 사이클 Git 체크포인트
- 명시적 완료 조건

다음 조건 중 하나라도 발생하면 자동 중단함.

- 같은 테스트가 2회 이상 동일 원인으로 실패함
- 동일 파일의 변경이 반복적으로 되돌아감
- diff가 사전에 정한 한도를 초과함
- 테스트 코드가 삭제되거나 기준이 완화됨
- 보안 또는 권한 정책 변경이 필요함
- Judge가 `revise`를 반복함
- Judge가 `stop_and_ask_user`를 반환함
- 사용자 입력이 필요한 상황이 발생함

## 5. 토론 시스템 규칙

### 5.1 독립 분석 후 토론함

초기 분석 단계에서는 다른 모델의 결론을 보기 전에 각 모델이 독립적으로 분석하도록 함.

권장 순서는 다음과 같음.

1. 각 모델이 프로젝트를 독립 분석함
2. 각 모델의 제안과 위험을 구조화함
3. 전체 제안을 서로 공개함
4. 각 모델이 다른 제안을 반박하거나 보완함
5. 역할과 책임 범위를 조정함
6. Judge가 근거와 검증 가능성을 평가함
7. 필요한 경우 사용자에게 선택을 요청함

초기부터 앞선 모델의 의견을 그대로 전달하면 뒤 모델이 첫 의견에 끌릴 수 있으므로 독립 분석 단계를 반드시 둠.

### 5.2 무조건 합의하지 않음

다음 행동을 금지함.

- 근거 없이 앞선 모델 의견에 동의함
- 다른 표현으로 같은 의견만 반복함
- 실제 검증 없이 “문제없음”이라고 결론 냄
- 다수 의견이라는 이유만으로 선택함
- 구현자가 자신의 구현을 최종 승인함

각 반론에는 최소한 하나 이상의 근거를 포함함.

- 코드 위치
- diff
- 테스트 결과
- 실행 로그
- 공식 문서
- 재현 절차
- 명확한 기술적 추론

### 5.3 토론 종료 조건

다음 조건을 모두 만족해야 토론을 종료함.

- 요구사항이 명확하게 정리됨
- 역할 충돌이 해소됨
- 구현 범위가 파일 또는 작업 단위로 정의됨
- 테스트 계획이 정의됨
- 위험 요소가 보고됨
- 사용자 판단이 필요한 항목이 없음
- Judge가 구현 진행 가능으로 판정함

사용자 판단이 필요한 항목이 하나라도 남아 있으면 구현하지 않고 질문함.

## 6. 역할 분담 규칙

역할은 파일 소유권이 아니라 **책임 영역**으로 정의함.

권장 역할 예시는 다음과 같음.

- Coordinator: 작업 순서, 충돌 조정, 사용자 보고 담당함
- Architect: 구조와 인터페이스 설계 담당함
- Backend: 서버, 상태, 데이터 처리 담당함
- Frontend: 대시보드와 사용자 상호작용 담당함
- QA: 테스트 계획, 회귀 검증, 재현 절차 담당함
- Reviewer: 코드 품질과 유지보수성 검토함
- Security: 권한, 비밀정보, 명령 실행 위험 검토함
- Judge: 증거를 기반으로 승인 여부 판정함

### 파일 범위 정책

파일 패턴은 강제 차단보다 경고와 검토 기준으로 우선 사용함.

다음 경우에만 강제 차단을 적용함.

- 민감한 설정 파일
- 비밀정보 파일
- 배포 설정
- 데이터베이스
- 사용자 데이터
- 프로젝트 외부 경로
- 사용자가 명시한 보호 파일

일반 코드에서는 다음 방식을 사용함.

- 주 작업자 지정함
- 검토자 지정함
- 다른 역할의 파일을 수정해야 하면 이유를 기록함
- 충돌 가능성이 있으면 담당 모델을 호출함
- 범위가 크게 바뀌면 사용자에게 질문함

## 7. 안전한 코딩 규칙

### 7.1 Git 체크포인트

실제 코딩 시작 전 반드시 다음 상태를 확인함.

- 현재 브랜치 확인함
- 미커밋 변경 확인함
- 사용자 작업 존재 여부 확인함
- 복구 가능한 체크포인트 생성함

사용자의 미커밋 변경이 있고 덮어쓸 위험이 있으면 즉시 중단하고 질문함.

각 구현 사이클 전후로 변경사항을 기록함.

### 7.2 금지 행동

사용자 승인 없이 다음 작업을 수행하지 않음.

- `git reset --hard`
- 강제 푸시
- 브랜치 삭제
- 대규모 파일 삭제
- 데이터베이스 초기화
- 사용자 파일 덮어쓰기
- 비밀키 출력
- 외부 서비스에 데이터 전송
- 관리자 권한 명령 실행
- 임의의 패키지 설치
- 테스트 삭제 또는 테스트 기준 완화
- 프로젝트 외부 경로 수정
- 위험 권한 옵션 추가 또는 확대

### 7.3 테스트를 기준에 맞추지 않음

코드를 테스트에 맞춰 정상적으로 수정해야 함.

다음 행동을 금지함.

- 실패 테스트 삭제함
- assertion을 약화함
- 테스트를 무조건 통과하도록 mock 처리함
- 오류를 삼켜 성공처럼 처리함
- 검증 명령을 실행하지 않고 통과했다고 보고함

테스트 자체가 잘못된 것으로 판단되면 수정하기 전에 사용자에게 근거를 보고하고 승인받음.

## 9. 상태 관리 개선

현재 전역 상태 의존성을 줄여야 함.

권장 방향은 다음과 같음.

- 세션 상태를 명시적 객체로 관리함
- 상태 전이를 함수 호출이 아닌 이벤트로 기록함
- 상태 저장은 원자적으로 수행함
- 세션 파일 저장 중 중단되어도 손상되지 않도록 임시 파일 후 교체함
- 실행 중인 프로세스와 저장된 상태를 구분함
- 재시작 시 실제 파일 상태와 세션 상태를 재검증함
- 하나의 세션에서 동시에 둘 이상의 쓰기 작업이 실행되지 않도록 잠금함

권장 주요 상태는 다음과 같음.

```text
IDLE
ANALYZING
DEBATING
WAITING_FOR_ROLE_ASSIGNMENT
WAITING_FOR_USER_APPROVAL
CODING
VALIDATING
REVIEWING
WAITING_FOR_USER_RESPONSE
COMPLETED
FAILED
CANCELLED
```

`WAITING_FOR_USER_RESPONSE` 상태에서는 다음 행동을 금지함.

- 자동 다음 턴 실행함
- 코딩 재개함
- 다른 에이전트 호출함
- 파일 수정함
- 승인 상태를 임의로 변경함

사용자가 답변한 뒤에만 명시적으로 다음 상태로 전환함.

## 10. 사용자 응답 대기 규칙

이 항목은 반드시 구현하고 테스트해야 함.

### 핵심 규칙

에이전트가 사용자 판단이 필요하다고 선언하면 다음과 같이 처리함.

1. 현재 실행 중인 새 코딩 작업을 시작하지 않음
2. 실행 중인 안전한 검증 명령은 종료 또는 완료 처리함
3. 세션 상태를 `WAITING_FOR_USER_RESPONSE`로 저장함
4. 질문과 선택지를 대시보드에 표시함
5. 자동 폴링은 유지하되 에이전트 턴은 실행하지 않음
6. 사용자의 답변을 세션에 기록함
7. 사용자의 답변을 관련 에이전트와 Judge에게 전달함
8. 기존 질문이 해결됐는지 Judge가 확인함
9. 해결된 경우에만 작업을 재개함

### 질문 감지 방식

자연어만으로 질문 상태를 판단하지 말고 구조화된 신호를 사용함.

권장 형식:

```json
{
  "action": "STOP_AND_ASK_USER",
  "reason": "기존 세션 저장 형식 변경 여부를 결정해야 함",
  "question": "기존 세션 자동 마이그레이션을 추가할지 선택해주기 바람.",
  "options": [
    {
      "id": "A",
      "label": "기존 형식 유지",
      "risk": "구현 복잡도 증가"
    },
    {
      "id": "B",
      "label": "자동 마이그레이션 후 새 형식 사용",
      "risk": "마이그레이션 코드 필요"
    }
  ],
  "recommended_option": "B",
  "blocking": true
}
```

`blocking: true`이면 사용자 응답 전까지 세션 진행을 금지함.

### 반드시 작성할 테스트

- 질문 상태에서 다음 에이전트가 실행되지 않음
- 서버를 재시작해도 질문 상태가 유지됨
- 사용자가 답변하기 전 파일 변경이 없음
- 일반 채팅 메시지와 질문에 대한 답변을 구분함
- 잘못된 선택값을 받으면 재질문함
- 질문 해결 후 정확한 단계에서 재개함
- 여러 질문이 동시에 발생하지 않도록 직렬화함
